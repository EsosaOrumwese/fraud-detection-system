## 4. KPI Set Gate

Before building any framework output, run a bounded KPI set gate.

The first profiling pass must answer:
- which controlled measures already exist in the Money and Pensions Service lane?
- which of those measures are stable enough to count as a bounded KPI set?
- what is the intended direction or measurement use for each?
- can one change-tracking reading be stated without inventing pre/post impact claims?
- does the resulting set stay compact enough to act as a framework surface rather than a loose list of metrics?

Required checks:
- continuity of the governed mixed-source lane
- continuity of the governed control surface
- continuity of the mixed-type and risk layers where they help explain measurement use
- ability to state one compact KPI set with intended-direction language

Those checks should be implemented as:
- aggregate-only or compact-output probes
- inherited compact outputs first
- no broad row materialisation into memory

Decision rule:
- if the KPI set remains explicit and controlled, proceed into framework and change-tracking materialisation
- if the measures are too unstable or ambiguous, narrow the slice before proceeding

## 8. Intended-Direction Strategy

The change-reading story has to stay honest.

The first-pass intended-direction strategy should answer:
- what exactly is the KPI set?
- what is the intended direction or use of each KPI?
- which part of the framework reading is observed current state and which part is intended direction?
- why does the interpretation stop at bounded progress language rather than stronger delivered-impact language?

Allowed progress posture:
- one bounded direction-of-travel reading
- one persistence-versus-progress reading
- one measured review posture if the evidence remains mixed

Disallowed progress posture:
- claiming delivered improvement impact the evidence does not support
- asserting full before/after success where the pack only supports directional interpretation
- importing unrelated improvement rhetoric as fake measurement proof

Decision rule:
- every retained KPI statement must earn its place in either:
  - framework definition
  - change tracking
  - improvement-assessment interpretation
- if a KPI statement does not support one of those purposes, drop it

## 9. Improvement-Assessment Strategy

This is what makes the slice useful rather than merely descriptive.

The first-pass interpretation layer should answer:
- what should a stakeholder conclude from the framework reading now?
- does the pack show stable pressure, early progress, or need for further review?
- what should be remeasured before any stronger outcome claim is made?

Required posture:
- use the framework to judge current status against intended direction
- use the change-tracking output to assess bounded progress or persistence
- use remeasurement language where the evidence is not strong enough for a more forceful claim

Disallowed posture:
- claiming completed improvement outcomes
- claiming enterprise framework ownership
- claiming service-delivery ownership beyond analytical support
