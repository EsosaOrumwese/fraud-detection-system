## 1. Requirement Being Targeted

The requirement being targeted here is the Money and Pensions Service `Data Insight Analyst` expectation around:
- supporting development and monitoring of `KPIs`
- working with quality or customer-experience frameworks
- measuring impact of improvement activity
- tracking change against intended outcomes

This maps directly to the candidate-profile requirement that the analyst should be able to sound like someone who:
- can work inside a defined measurement framework
- can make `KPIs` usable rather than merely listing them
- can track movement over time
- can help stakeholders assess whether change is moving in the intended direction

This note exists because that requirement is easy to flatten into the vague statement:
`I support KPIs and measurement`

That is not the real burden.

The employer is asking for something sharper:
- one bounded framework-measurement surface
- one explicit KPI set tied to that framework
- one change-tracking or progress reading
- one interpretation of what the movement means for improvement assessment

## 2. The Analogue On The Current Platform

The current platform does not contain a literal Money and Pensions Service `CX&Q` measurement framework, formal outcome scorecard, or full improvement-review board, so the slice has to use an honest analogue.

The closest truthful analogue is:
- one mixed-source reporting lane already established in Money and Pensions Service `3.B`
- one governed output surface already established in Money and Pensions Service `3.A`
- one mixed-type interpretive layer already established in Money and Pensions Service `3.D`
- one risk-and-likely-driver layer already established in Money and Pensions Service `3.E`
- one KPI-and-improvement methodological posture already proven in Hertfordshire `01` and `04`

So for this slice:
- `framework measurement` maps to `one bounded set of measures fixed over the existing Money and Pensions Service lane`
- `supporting KPIs` maps to `one explicit KPI set whose purpose, reading, and change logic are made clear`
- `tracking change against intended outcomes` maps to `one compact progress reading that distinguishes current status from intended direction`
- `measuring improvement activity` maps to `one bounded interpretation of whether the framework indicates progress, persistence, or need for further review`

In practice, that means the default posture should be:
- `intended-direction` first
- `progress reading` second
- stronger `impact of improvement activity` language only if the evidence behaves like a real before/after measurement rather than a directional review surface

The claim must therefore remain:
- framework-measurement shaped
- KPI-support shaped
- progress-tracking shaped
- platform-truth bounded
- careful not to imply ownership of a full enterprise measurement strategy or the complete Money and Pensions Service framework where this slice only proves one bounded framework-measurement pack

## 7. Execution Posture

Execution should stay:
- framework-first
- KPI-definition-first
- change-tracking-first
- bounded-improvement-assessment-first

Execution should not drift into:
- generic reporting language with no framework substance
- fake enterprise scorecard ownership
- broad improvement-outcomes claims
- reusing Hertfordshire wording so directly that the Money and Pensions Service lane disappears

The first pass should:
- start from the existing controlled Money and Pensions Service signal set
- test whether one compact KPI set can be fixed as a framework surface
- test whether one change-tracking reading can be stated from the inherited lane
- only then write the improvement-assessment interpretation in practical language

That execution order matters because:
- the measures must exist before their change is interpreted
- the change-tracking reading must be explicit before any improvement-assessment language is written
- readability should only be tightened after the framework logic and limits are fixed

If the pack only supports directional progress language rather than measured impact, keep it that way.
This slice does not need a strong causal improvement-outcome claim to be valid.
