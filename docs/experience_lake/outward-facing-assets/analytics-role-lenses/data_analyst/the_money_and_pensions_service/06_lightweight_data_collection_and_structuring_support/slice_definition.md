## 1. Requirement Being Targeted

The requirement being targeted here is the Money and Pensions Service `Data Insight Analyst` expectation around:
- supporting new data-collection mechanisms
- supporting lightweight practical solutions for information capture or assessment
- improving how evidence is structured so that reporting, investigation, and measurement work become easier

This maps directly to the candidate-profile requirement that the analyst should be able to sound like someone who:
- can improve the way small parts of the evidence are captured or assembled
- can support lightweight practical data solutions
- can make information easier to use downstream
- can do that without needing a full product or platform build

This note exists because that requirement is easy to flatten into the vague statement:
`I supported data collection`

That is not the real burden.

The employer is asking for something sharper:
- one bounded mechanism or structuring surface
- one clear downstream use benefit
- one lightweight practical solution rather than a broad technical estate

## 2. The Analogue On The Current Platform

The current platform does not contain a literal Money and Pensions Service feedback platform, survey application, or live collection product, so the slice has to use an honest analogue.

The closest truthful analogue is:
- one mixed-source reporting lane already established in Money and Pensions Service `3.B`
- one governed output surface already established in Money and Pensions Service `3.A`
- one mixed-type evidence lane already established in Money and Pensions Service `3.D`
- one framework-measurement surface already established in Money and Pensions Service `3.F`
- one additional lightweight evidence-structuring mechanism that standardises how a downstream narrative or issue surface can be captured and reused

So for this slice:
- `new data-collection mechanism` maps to `one bounded lightweight capture or structuring surface added to the analytical lane`
- `lightweight solution` maps to `one compact reusable mechanism that makes downstream reporting, investigation, or measurement easier`
- `practical use` maps to `one explicit note showing what the mechanism enables downstream`

In practice, that means the default posture should be:
- `evidence-structuring` first
- `standardised capture surface` second
- stronger `new data-collection mechanism` language only if the slice really adds a new capture point rather than only formalising and standardising an existing one

The claim must therefore remain:
- lightweight-mechanism shaped
- evidence-structuring shaped
- downstream-use shaped
- platform-truth bounded
- careful not to imply a full collection product, survey platform, or application estate where this slice only proves one bounded practical mechanism

- support for new collection mechanisms

## 7. Execution Posture

Execution should stay:
- mechanism-first
- standardisation-first
- downstream-use-first
- bounded-solution-first

Execution should not drift into:
- generic reporting language with no mechanism substance
- fake product or application ownership
- broad collection-platform claims
- decorative interface language with no analytical use consequence

The first pass should:
- start from the existing controlled Money and Pensions Service lane
- identify one part of the current lane that would benefit from a lightweight capture or structuring surface

That execution order matters because:
- the mechanism must exist before its usefulness is claimed
- the standardised surface must be explicit before any downstream-use language is written
- readability should only be tightened after the mechanism logic and limits are fixed

If the result is only a standardised structuring layer rather than a genuinely new capture point, keep the claim at that level.
This slice does not need a stronger collection-platform claim to be valid.
