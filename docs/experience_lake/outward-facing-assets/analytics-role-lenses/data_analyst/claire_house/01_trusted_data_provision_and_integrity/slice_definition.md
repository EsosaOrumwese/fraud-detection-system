## 1. Requirement Being Targeted

The requirement being targeted here is the Claire House `Data & Insight Analyst` expectation around:
- supporting the production of organisational data provision
- supporting the management of organisational data provision
- supporting the protection of organisational data provision
- supporting the integrity of organisational data provision

This maps directly to the Claire House candidate-profile requirement that the analyst should be able to:
- help maintain reliable organisational data provision across systems
- support data integrity and quality processes
- contribute to the protection and proper use of information
- work as more than a passive report consumer by helping keep the underlying information environment dependable

This note exists because that requirement is easy to flatten into the vague statement:
`good with data quality`

That is not the real burden.

The employer is asking for something broader and more organisational:
- trusted data provision
- integrity-aware handling
- protection-conscious use
- reliability of the analytical information environment

## 2. The Analogue On The Current Platform

The current platform does not contain a literal charity healthcare organisational estate with live internal and external operational systems, so the slice has to use an honest analogue.

The closest truthful analogue is:
- one bounded multi-surface analytical provision lane
- one maintained detailed dataset under that lane
- one protected reporting-safe output above it
- one explicit integrity and control layer showing how trustworthy provision is maintained

So for this slice:
- `production, management, protection, and integrity of data provision` in the employer wording maps to `trusted analytical data provision` on the current platform
- the proof burden is:
  - define the data-provision path
  - identify where trust could be lost
  - control the path through validation, reconciliation, and authority rules
  - show that the protected output depends on that controlled provision layer

The claim must therefore remain:
- organisational-data shaped
- platform-truth bounded
- careful not to imply enterprise MDM or a full information-governance function where this slice only proves one bounded trusted data-provision lane

## 9. Lens-by-Lens Execution Checklist

### 9.1 Lens 03 - Control the trusted data-provision lane

This is the core owner.

Tasks:
1. Define the bounded provision lane explicitly:
- reporting window
- maintained dataset grain
- protected downstream output
- source contribution path
2. Map source contribution:
- which source surfaces enter the lane
- which joins or handoff steps sit between them
- which fields are carried downstream
3. Profile the provision lane before any widening:
- row counts
- period coverage
- key coverage
- null rates for required fields
- duplicate checks at the maintained grain
4. Define authority and protection rules:
- authoritative key
- authoritative field choices
- fields safe for downstream provision
- fields that remain control-only or caveated
5. Build integrity checks:
- completeness
- consistency
- grain integrity
- source-to-provision alignment
6. Record one provision-risk note:
- where trust could be lost
- what was checked
- what was controlled
- what remains caveated

### 9.2 Lens 02 - Show the protected downstream dependency

This is the main support lens.

Tasks:
1. Build one protected downstream summary or reporting-safe output from the controlled provision lane.
2. Show that the downstream output depends on the controlled provision definition.
3. Record one protection note:
- which downstream use is being protected
- what would be unsafe in a loose raw-extract path
- what analytical trust boundary the controlled lane protects
4. Keep the proof compact:
- one protected output
- one dependency note
- no dashboard sprawl

### 9.3 Lens 09 - Make provision integrity repeatable

This is what turns the slice from one careful control exercise into maintainable organisational-style data provision support.

Tasks:
1. Stabilise the lane definition:
- provision path
- maintained grain
- authoritative fields
- allowed exclusions
- release-safe output rule
2. Version the control logic:
- profiling logic
- integrity checks
- reconciliation checks
- protected output logic
3. Create one run checklist:
- what to profile first
- what checks must pass
- what breaks comparability
- what output is safe to release
4. Document caveats:
- what the lane is suitable for
- what it should not be used for
- what source changes would invalidate comparability

### 9.4 Lens 05 - State why the lane matters to organisational use

This is light support only.

Tasks:
1. Write one short note explaining why controlled provision matters for organisational reporting or insight use.
2. Keep it bounded:
- no broad change programme claim
- no stakeholder-discovery sprawl
- one concise organisational-use justification only
