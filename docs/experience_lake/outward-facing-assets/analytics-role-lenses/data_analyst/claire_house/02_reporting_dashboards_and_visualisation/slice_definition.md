## 1. Requirement Being Targeted

The requirement being targeted here is the Claire House `Data & Insight Analyst` expectation around:
- using business-intelligence-style outputs and statistical techniques to provide ad hoc reports
- using business-intelligence-style outputs and statistical techniques to provide scheduled reports
- producing dashboards and data selections for the organisation
- developing interactive or visual outputs that enable insight
- tracking key performance indicators through visualisation and reporting outputs

This maps directly to the Claire House candidate-profile requirement that the analyst should be able to describe:
- building dashboards, visualisations, and reports that different parts of the organisation can use
- turning raw or governed data into scheduled and ad hoc reporting outputs
- creating KPI-led insight products rather than isolated extracts
- delivering reporting outputs that are structured, reusable, and useful to organisational stakeholders

This note exists because that requirement is easy to flatten into the vague statement:
`good at dashboards`

That is not the real burden.

The employer is asking for something broader and more useful:
- scheduled and ad hoc reporting delivery
- dashboard and visualisation outputs
- KPI tracking through visual reporting
- organisational usability rather than analyst-only chart production

## 8. Lens-by-Lens Execution Checklist

### 8.1 Lens 01 - Fix the KPI logic

This is the performance-meaning layer that stops the slice from turning into decorative charts.

Tasks:
1. Define one bounded KPI family:
- volume or intake
- progression or conversion
- one downstream quality or yield signal
- one segment or band dimension for comparison
2. Fix the reporting window:
- one bounded month
3. Define one supporting comparison:
- summary against overall
- or supporting band-level contrast
4. Write one short `what this KPI set is for` note:
- what should be read at summary level
- what should be read in the supporting detail cut

### 8.2 Lens 02 - Build the reporting and visualisation outputs

This is the core owner.

Tasks:
1. Define the reporting product:
- one summary page
- one ad hoc supporting detail page or cut
2. Build one summary output:
- headline KPI view
- one compact comparison view
- one visual layout suitable for organisational reading
3. Build one ad hoc supporting detail output:
- one more detailed cut using the same KPI definitions
- one output that shows the same truth in a slightly deeper view
4. Keep KPI meaning stable:
- same names
- same definitions
- same comparison basis
5. Ensure the outputs look like bounded reporting products:
- not loose charts
- not a dashboard estate
- one compact pack only

### 8.3 Lens 09 - Make the outputs repeatable and controlled

This is what turns the slice from one reporting example into a believable reporting responsibility.

Tasks:
1. Stabilise KPI definitions:
- names
- logic
- comparison basis
- supporting detail logic
2. Version the build logic:
- KPI shaping logic
- summary output logic
- supporting detail logic
3. Create one run checklist:
- what inputs are needed
- what checks must pass
- what output is safe to release
4. Document caveats:
- what the reporting pack answers
- what it does not answer
- what would break comparability

### 8.4 Lens 08 - Keep the outputs readable and usable

This is light support only.

Tasks:
1. Write one short audience-usage note:
- what a general organisational reader should look at first
- what the supporting detail view adds
2. Add one short explanation note if needed:
- how to read the summary and detail together
3. Keep this bounded:
- no broad briefing pack
- no storytelling sprawl
