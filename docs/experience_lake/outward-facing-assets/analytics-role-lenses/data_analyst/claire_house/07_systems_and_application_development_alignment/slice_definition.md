## 1. Requirement Being Targeted

The requirement being targeted here is the Claire House `Data & Insight Analyst` expectation around:
- collaborating with the `Application Development Team`
- aligning data initiatives with systems development
- ensuring seamless integration of data pipelines and storage solutions

This maps directly to the Claire House candidate-profile requirement that the analyst should be able to describe:
- working comfortably alongside developers or systems teams
- aligning analytical data work with systems or application changes
- supporting data-pipeline or storage integration in a bounded analytical context
- understanding that reporting quality depends on good upstream and downstream system alignment

This note exists because that requirement is easy to flatten into the vague statement:
`comfortable working with technical teams`

That is not the real burden.

The employer is asking for something sharper:
- identify a systems-facing boundary that matters to the analytical lane
- make the interface and handoff expectations explicit
- show what downstream analytical risk appears if that alignment is weak
- show one bounded alignment control that protects the lane

## 2. The Analogue On The Current Platform

The current platform does not contain the literal Claire House application-development team, application estate, or live systems-integration programme, so the slice has to use an honest analogue.

The closest truthful analogue is:
- one governed Claire House analytical lane already established through `3.A`, `3.D`, and `3.F`
- one upstream/downstream system-facing boundary that the lane depends on, especially the handoff from provision-safe maintained data into regenerated downstream outputs
- one explicit handoff or interface contract for that boundary
- one alignment risk if the boundary is loose or misunderstood
- one bounded control that protects downstream analytical outputs

So for this slice:
- `collaborate with the Application Development Team` maps to `work at a systems-facing interface where analytical outputs depend on clear upstream/downstream handoff`
- `align data initiatives with systems development` maps to `make interface assumptions, input/output expectations, and dependency boundaries explicit`
- `ensure seamless integration of data pipelines and storage solutions` maps to `show that the maintained analytical lane and the regeneration path consuming it remain safe when the handoff boundary is explicitly controlled`

This means the slice can honestly support:
- interface and handoff alignment
- bounded data-pipeline integration support
- explicit dependency control around a systems-facing boundary
- protection of downstream analytical outputs through that alignment

It does not mean the slice can honestly support:
- application-development ownership
- full integration-platform design
- organisation-wide systems-integration architecture

The claim must therefore remain:
- interface-alignment shaped
- governed-lane bounded
- careful not to imply that one bounded handoff proof equals a full systems-development estate
