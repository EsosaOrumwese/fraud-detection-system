## 1. Requirement Being Targeted

The requirement being targeted here is the Claire House `Data & Insight Analyst` expectation around:
- monitoring data quality
- auditing data quality
- making recommendations for data quality improvements where appropriate
- assisting the `Data & Insight Manager` with data governance tasks
- assisting the `Data & Insight Manager` with `IG` governance tasks

This maps directly to the Claire House candidate-profile requirement that the analyst should be able to describe:
- actively checking data quality rather than only using it
- auditing analytical outputs and the controls behind them
- identifying weaknesses in data capture or analytical quality
- recommending improvements that strengthen trust in organisational reporting
- supporting governance and `IG` work in a bounded but credible way

This note exists because that requirement is easy to flatten into the vague statement:
`good at data quality`

That is not the real burden.

The employer is asking for something more deliberate:
- active quality monitoring
- explicit audit posture
- recommendation-making from findings
- governance-support discipline

## 2. The Analogue On The Current Platform

The current platform does not contain a literal Claire House organisational governance office, board-approved quality register, or full `IG` operating function, so the slice has to use an honest analogue.

The closest truthful analogue is:
- one trusted analytical lane already established in Claire House `3.A`
- one maintained analytical layer already established in Claire House `3.D`
- one compact quality-monitoring and audit pack over that same bounded lane
- one recommendation set that follows directly from the audit findings
- one governance-support note tied to controls, caveats, and safe downstream use

So for this slice:
- `monitoring data quality` maps to `repeatable quality checks across the bounded analytical lane`
- `auditing data quality` maps to `an explicit audit findings pack over the provision and maintained layers`
- `making recommendations` maps to `finding-to-action improvement notes`
- `assisting with governance and IG tasks` maps to `bounded support for control, caveat, and responsible data-use posture`

This means the slice can honestly support:
- active quality monitoring
- explicit audit and findings
- recommendations linked directly to findings
- governance-support material that helps the `Data & Insight Manager` in a bounded way

It does not mean the slice can honestly support:
- formal ownership of a governance function
- formal ownership of an `IG` function
- enterprise-wide quality administration across the whole organisation

The claim must therefore remain:
- quality-and-governance shaped
- platform-truth bounded
- careful not to imply ownership of a full governance programme where this slice only proves one bounded analytical-lane audit and recommendation pack

- confirm the maintained analytical layer and control posture from Claire House `3.D`
- define the audit scope over that bounded lane
- materialise one quality-monitoring summary and one audit findings output

The execution substrate should be:
- the existing controlled provision lane already established in Claire House `3.A`
- the maintained analytical layer already established in Claire House `3.D`
- compact audit outputs rather than broad raw rescans

- the work should sound like active quality monitoring and governance support, not like generic governance rhetoric

Memory and query discipline:

- keep heavy work in the query engine
- no broad raw pandas loads

- if an extra probe is needed, filter period and fields at scan time and stop once the audit-and-recommendation question is answered

## 8. Lens-by-Lens Execution Checklist

### 8.1 Lens 03 - Build the quality-monitoring and audit layer

This is the core owner.

Tasks:
1. Define the audit scope explicitly:
- trusted provision lane
- maintained analytical layer
- protected downstream use
2. Build one quality-monitoring summary:
- key checks in scope
- current pass/fail or risk status
- bounded lane coverage
3. Build one audit findings output:
- where the lane is strong
- where quality risk remains
- which weakness matters most
4. Record one governance-support boundary:
- what control posture this supports
- what it does not yet prove

### 8.2 Lens 04 - Ground the audit in the maintained layer

This is light support, but it matters.

Tasks:
1. Tie the audit findings explicitly to the maintained analytical layer where relevant.
2. Show which earlier control improvement is being monitored.
3. Keep it bounded:
- no fresh infrastructure claim
- one maintained-layer dependency only

### 8.3 Lens 09 - Make the audit repeatable

This is what turns the slice from a quality comment into an audit-support proof.

Tasks:
1. Stabilise the audit logic:
- checks
- thresholds or pass logic
- audit output structure
- rerun logic
2. Version the audit pack:
- monitoring query or transformation logic
- findings logic
- release or checklist logic
3. Create one audit checklist:
- what must be checked
- what must be documented
- what would invalidate comparability
4. Document caveats:
- what the audit answers
- what it does not answer
- what would require a broader governance review

### 8.4 Lens 05 - Turn findings into recommendations

This is support, but it is essential to the responsibility.

Tasks:
1. Write one recommendation set linked directly to the findings.
2. Keep recommendations bounded:
- no enterprise transformation plan
- no broad governance programme
- one concise findings-to-action chain
3. Write one governance-support note:
- what the Data & Insight Manager could use from this audit pack
- how the pack supports governance or `IG` work without claiming ownership of it
