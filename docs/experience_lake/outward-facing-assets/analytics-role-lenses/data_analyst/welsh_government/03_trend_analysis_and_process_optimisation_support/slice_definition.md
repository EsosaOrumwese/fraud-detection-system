## 1. Requirement Being Targeted

The requirement being targeted here is the Welsh Government `Payroll Data Analyst` expectation around:
- analysing recurring operational or payroll-style data for meaningful trends and patterns
- using those trends to identify process inefficiency, avoidable friction, or repeated error pressure
- supporting improvement or optimisation of systems and processes from evidence
- helping operational stakeholders focus review attention on the part of the cycle most worth improving first

This maps directly to the candidate-profile requirement that the analyst should be able to sound like someone who:
- does more than validate outputs after the fact
- can recognise recurring operational patterns rather than one-off defects
- can translate those patterns into practical process-improvement or optimisation support
- can make evidence useful for follow-through without claiming to own the whole transformation effort

This note exists because that responsibility is easy to flatten into the vague statement:
`good at process improvement`

That is not the real burden.

The employer is asking for something sharper:
- identify repeated operational patterns
- explain what those patterns imply for process or systems performance
- point to where review or optimisation attention should go first
- support improvement with evidence that remains proportional to the underlying control lane

The current platform does not contain a literal payroll-transformation programme, live payroll software estate, or end-to-end public-sector process-optimisation office, so the slice has to use an honest analogue.

The closest truthful analogue is:
- the recurring control lane already established in Welsh Government `A + D`
- the review-ready, traceable release story already established in Welsh Government `3.E`
- the repeated denominator-drift pattern that remains stable across both retained cycles

So for this slice:
- `trend analysis` in the employer wording maps to `reading recurring movement and persistence across the current-plus-prior control window`
- `process or system optimisation` maps to `bounded optimisation-support interpretation over the recurring discrepancy pattern`
- the proof burden is:
  - show that the control issue is recurring rather than incidental
  - show what operational inefficiency or avoidable rework it implies
  - support one targeted review or optimisation recommendation
  - stop short of claiming that the wider process has already been redesigned or fixed

The claim must therefore remain:
- trend-shaped
- improvement-support shaped
- optimisation-support shaped
- platform-truth bounded
- careful not to imply a full payroll-transformation estate where this slice only proves one bounded process-optimisation support pack

Default first-pass posture:

- targeted optimisation-support recommendation third
- delivered-gain language only if the bounded evidence genuinely supports it
- broader process-redesign language only where the bounded evidence genuinely supports it

## 9. Lens-by-Lens Execution Checklist

### 9.1 Lens 01 - Fix the recurring trend reading

This is the core owner.

Tasks:
1. Confirm the recurring control window explicitly.
2. State what stayed the same and what repeated across both retained cycles.
3. Show whether the discrepancy pattern is persistent enough to count as a repeated operational signal.
4. Keep the reading bounded to the same recurring lane rather than broad process speculation.

### 9.2 Lens 05 - Turn the repeated pattern into optimisation-support meaning

This is the main support owner.

Tasks:
1. Define the improvement or optimisation question clearly.
2. Translate the repeated discrepancy into one avoidable-rework or process-friction interpretation.
3. State one targeted review or optimisation recommendation.
4. Keep the recommendation proportional to the evidence and below full redesign ownership.

### 9.3 Lens 02 - Package the evidence cleanly

Tasks:
1. Materialise one compact recurring trend summary.
2. Materialise one optimisation-support output.
3. Keep naming stable between the inherited control pack and the new optimisation pack.
4. Make the outputs readable as decision-support rather than dense control tables only.

### 9.4 Lens 08 - Keep the recommendation usable

Tasks:
1. Write one short action-facing note.
2. Make the targeted review recommendation legible to operational stakeholders.
3. Avoid drifting into consultancy or broad transformation wording.

### 9.5 Lens 03 - Escalate only if trust ambiguity reappears

Tasks:
1. Check whether the repeated pattern still rests safely on the earlier trusted control family.
2. Record one trust caveat only if necessary.
3. Avoid turning the slice back into a trust or audit slice unless the trend itself becomes unstable.
