## 4. Control-Lane Gate

Before building any Welsh Government output, run a bounded control-lane gate.

The first profiling pass must answer:
- which inherited compact lane gives the cleanest validation or balancing analogue?
- can one explicit discrepancy class be stated cleanly from that lane?
- can that discrepancy class be turned into one reconciliation-ready surface?
- does the resulting pack remain a recurring operational-control proof rather than a broader payroll-function story?
- does the resulting control pack stay compact enough to act as a bounded proof object?

Required checks:
- continuity of the inherited discrepancy logic
- continuity of the inherited governed control surface
- ability to state one balancing or validation rule explicitly
- ability to state one recurring-support consequence from the same pack

Those checks should be implemented as:
- aggregate-only or compact-output probes
- inherited compact outputs first
- no broad row materialisation into memory

Decision rule:
- if the control lane remains explicit and compact, proceed into validation, discrepancy, and reconciliation materialisation
- if the problem is too broad or ambiguous, narrow the slice before proceeding

## 9. Recurring-Support Strategy

This is what makes the slice useful rather than merely technical.

The first-pass recurring-support layer should answer:
- what recurring operational activity becomes safer or clearer because of the control pack?
- does the pack improve confidence, repeatability, or balancing clarity?
- what should an operator or manager now be able to rely on more safely?

Required posture:
- use the control pack to improve one explicit recurring operational use
- use the reconciliation surface to make that use clearer or safer
- keep the consequence proportional to the evidence

Disallowed posture:
- claiming whole payroll-cycle ownership
- claiming wide statutory compliance ownership
- claiming broader organisational benefits not evidenced by the pack
