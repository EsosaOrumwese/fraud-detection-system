After the harness or source under-drive was corrected with the stronger evidence-backed WSP posture, the path stopped being a proving-shape story and started being a true runtime story. First, the path exposed `IG_UNHEALTHY:BUS_UNHEALTHY` because the live Lambda was still using `IG_HEALTH_BUS_PROBE_MODE=describe`. After repinning that to `none`, the run stayed clean on `4xx` and `5xx`, but throughput and latency still missed badly. Then the `900`-concurrency probe exposed something worse: `IG_PUSH_REJECTED`, `PUBLISH_AMBIGUOUS`, and concrete `KAFKA_PUBLISH_TIMEOUT` under valid traffic, which made it clear that just raising concurrency was not a valid bridge. This is the episode where the path became runtime-critical rather than just route-correct.

4. `Telemetry hardening`

The next material posture change was observability. A timing-telemetry deployment initially failed because the helper itself was defective, so one run had to be rejected as invalid proof. After that was fixed and redeployed, the path gained a new kind of evidence: it could now separate cold-fleet gate initialization from hot-path admission cost. The rerun showed the active red was no longer throttling, but a more precise family: large gate-init cost plus substantial admit-path time. That matters because the boundary changed from red and somewhat opaque to red, but now attributable.

5. `Store-health semantics were corrected`

Another posture-changing episode was the object-store health correction. Fresh bounded runs showed benign governance append contention (`S3_APPEND_CONFLICT` / `ConditionalRequestConflict`) was being misclassified as object-store unhealth, causing the ingress path to fail closed with `IG_UNHEALTHY:OBJECT_STORE_UNHEALTHY`. Once that exception family was broadened to treat the live shapes as benign, reruns became semantically clean and landed only just below target, with `4xx = 0`, `5xx = 0`, and no Lambda errors or throttles. That changed the path from runtime still sick to runtime now semantically clean, with the active blocker narrowed to the proof surface.

6. `Methodology correction`

At that point the work changed from runtime tweaking to proof-boundary discipline. The notes explicitly reject chasing greens through synchronized starts or warmed single-bin reshapes. Instead, the accepted posture became: freeze the semantically trustworthy unsynchronized baseline, tighten telemetry on that same baseline, and only then judge repeatability. That is a real posture change for this path, because it defines how the boundary is allowed to earn readiness: not through convenient reshapes, but through one frozen truthful baseline.

7. `APIGW-side telemetry lock`

The next decisive shift was closing the APIGW telemetry gap materially: route-level detailed metrics were enabled, the APIGW access-log group was created, the CloudWatch Logs resource policy was added, and real access-log delivery from the active API and stage was verified. That gave the path its first truthful per-request external evidence surface on the exact boundary being proved. This matters because before that, the boundary could be semantically clean but still judged through coarser or lagging telemetry. After this episode, the path could be judged on the exact edge it claimed to own.

8. `Exact-window scoring repin`

Once APIGW access logs were live, the old minute-aligned APIGW metric gate could be shown to be the wrong proof boundary. Repeated runs were semantically clean, but the minute-bin gate still rendered them red at `2999.x eps`. Exact access-log windows on confirmed participation showed the path was actually above target (`3023.950 eps`, `3024.642 eps`) on the exact external boundary. This is the episode where the path's readiness posture changed from almost green on a coarse metric to truthfully green on the right external proof surface.

9. `Drift and cost cleanup`

After the proof boundary was repinned, the last big posture change was to reconcile away retained drift: the obsolete ingress ECS / ALB / SSM surface was destroyed, the stale SSM service URL was removed, the ingress ECS cluster became inactive, the APIGW telemetry surface was codified in Terraform, and single ownership was restored for the log policy. This matters because it changed the path from one true boundary, but with retained internal ingress drift still hanging around, to one true boundary, one true telemetry story, and no silent internal alternative left in the accepted live network.

10. `Closure authority`

The final posture change is the closure itself. The accepted cost posture held on the frozen truthful baseline with exact APIGW access-log scoring: `3018.458 eps`, `4xx = 0`, `5xx = 0`, low latency, and no open blockers on the accepted baseline. The final phase authority then records `Phase 0` as proven on the exact APIGW-scored closure basis, with steady `3025.30 eps`, burst `6019.50 eps`, recovery `3019.21 eps`, zero `4xx` or `5xx`, zero Lambda errors or throttles, flat DLQ delta, and recovery to sustained green at `0 s`. That is the point where the path stops being hardening in progress and becomes part of the accepted production-ready base.

`What this mapping says in one sentence`

`Boundary access path` did not become production-ready because one endpoint existed and later went green. It became production-ready because the work progressively removed boundary drift, harness under-drive, misleading runtime probes, telemetry blindness, proof-boundary distortion, and retained internal-ingress ambiguity until one explicit APIGW front door could be judged and closed on the exact surface it claimed to own.
