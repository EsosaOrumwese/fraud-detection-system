That was the right bridge because it did not simplify the boundary; it made the boundary honest. The readiness property improved here was boundary truth: one real external front door, not a convenient family of possible ingress surfaces.

`Episode 2 - once the right edge was forced, the first red was a proving-shape defect, not a broken edge`

The first fresh bounded run on the corrected execute-api edge was red, but in a very revealing way: `4xx = 0`, `5xx = 0`, flat DLQ, zero Lambda errors or throttles, healthy latency, and yet admitted throughput only reached about `1734 eps`. System-design-wise, this matters because it separated two different concerns: whether the front door was broken versus whether the proving source was actually generating production-shaped pressure. The answer was that the path was cleanly admitting what it received, but the current proving shape was not yet generating the declared load. So this was a proof-shape or harness defect, not a runtime failure on the boundary itself. The accepted bridge was to repin WSP onto the stronger already-proven launch posture instead of remediating ingress blindly. That improved attribution truth: the path was no longer being blamed for a source-generation failure.

`Episode 3 - after the harness was corrected, the path exposed a real runtime red, and a naive scale-up made it worse`

Once the stronger WSP shape removed the under-drive ambiguity, the next runs showed the first genuine runtime problems on this path. First, the boundary was returning `IG_UNHEALTHY:BUS_UNHEALTHY` because the live Lambda still had `IG_HEALTH_BUS_PROBE_MODE=describe`; once that was repinned to `none`, the path reached a more truthful red: `4xx = 0`, `5xx = 0`, but only about `1584 eps` and very high ingress latency. Then the `900`-concurrency probe made the key engineering point: pushing more concurrency into the same path did not solve throughput, it produced `KAFKA_PUBLISH_TIMEOUT -> PUBLISH_AMBIGUOUS`, quarantine receipts, and fail-closed publish ambiguity on valid traffic. That means the right reading of this episode is that the edge was now operationally insufficient under the declared envelope, and a crude scale-up would have falsified readiness by making semantics worse. This was a true runtime defect. The accepted bridge was therefore not more concurrency, but keeping the semantically clean `600`-concurrency baseline, improving diagnostics, and treating the hot path itself as the active boundary. That improved semantic cleanliness under pressure, not just raw throughput.

`Episode 4 - the next red was in observability itself, and the path could not be judged honestly until that was fixed`

The next important turn was that the first timing-telemetry deployment itself introduced a new red: the helper had a `NameError`, which meant the rerun was invalid proof. After repairing and redeploying the timing helper, the same path became newly readable: the logs showed gate initialization around `~1.4 s`, successful admit-path work still costing `~0.8-0.9 s`, and some long-running duplicate or in-flight cases taking `11-15 s` before quarantining. So this episode is really two things: first a proof-boundary or observability defect, then an observability bridge that made the path diagnosable on its own terms. The accepted bridge was right because it did not change ingress semantics at all; it only removed blindness around them. The readiness property improved here was explainability of the edge under pressure.

`Episode 5 - the path was failing closed for the wrong reason, because adjacent store contention was being misread as boundary sickness`

The next meaningful red was not that ingress could not handle traffic, but that ingress was misclassifying a neighboring failure family. Fresh runs showed benign governance append contention on the object-store side (`S3_APPEND_CONFLICT` / `ConditionalRequestConflict`) being interpreted as `IG_UNHEALTHY:OBJECT_STORE_UNHEALTHY`, which forced the edge to fail closed on valid traffic for the wrong reason. That is a very different class of problem. System-design-wise, it means the boundary health logic was not yet accurately partitioning local ingress truth from adjacent governance or storage contention. This is a semantic health-classification defect. The accepted bridge was to broaden the benign exception family so `S3_APPEND_CONFLICT`, `ConditionalRequestConflict`, and `PreconditionFailed/412` no longer poisoned the ingress health signal. That was the right bridge because it did not relax failure handling generally; it made fail-closed behavior truthful to the actual boundary. The evidence that posture changed is strong: the reruns immediately after this were semantically clean at essentially target rate, with `4xx = 0`, `5xx = 0`, and the remaining blocker narrowed back to the proof surface rather than the runtime surface.

`Episode 6 - after runtime semantics were clean, the active blocker moved to the proof boundary itself`

Once the edge was semantically clean, the remaining near-miss looked tiny and persistent under the old minute-aligned APIGW metric gate. The full lane logs showed the miss was concentrated in the first measured minute, while the second measured minute was effectively on target. The synchronized-start correction and the warmed single-bin correction both failed as valid baselines, which matters because it shows the work resisted the easy but wrong move of just finding a shape that goes green. The methodological shift was to freeze the semantically trustworthy unsynchronized baseline and tighten the proof boundary around that same baseline instead of reshaping the run. That is the critical systems-design move for this path: the question changed from runtime capacity to scoring truth. This episode is therefore a proof-boundary defect, not a runtime defect. The accepted bridge had two parts: first, make the APIGW side materially observable by enabling detailed metrics and access logs; second, score steady-state through the exact APIGW access-log window anchored on confirmed participation rather than the slightly distorted minute-bin view. The evidence is decisive: the exact APIGW window on repeated runs showed `3023.950 eps` and `3024.642 eps` with clean latency on the same supposedly red baseline. Later, with the accepted cost posture held constant, the exact-window gate closed cleanly at `3018.458 eps`. That is a real posture change: the path moved from almost green on a coarse surface to actually green on the edge it claims to own.

`Episode 7 - even after the path was truthfully green, retained ingress drift still had to be removed from the accepted story`

The last important episode for this path is that the work did not stop at a green receipt. The notes show the readiness graphs were still overclaiming the retained internal ALB / ECS ingress surface as if it were still part of the accepted external boundary, and the live stack still carried the obsolete internal ingress resources and stale SSM URL. This is not a throughput issue at all; it is a drift or ownership defect. The accepted bridge was to repin the live Lambda artifact safely, codify APIGW metrics and access logs in Terraform, destroy the retained ALB / ECS / SSM ingress path, and regenerate the graphs so they matched the new truth. That was the right bridge because it made the final ready posture single-owned: one live ingress edge, one telemetry story, one accepted graph story. The evidence after reconcile is exactly what this path needed: `IG_SERVICE_ENABLED = false`, no stale SSM service URL, ingress ECS cluster inactive, APIGW detailed metrics enabled, and access logs attached under single ownership. That is the moment the path stopped being merely green enough and became production-ready in a governable way.
